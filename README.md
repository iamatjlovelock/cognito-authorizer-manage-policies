# Manage Policies Demo

A demo application that demonstrates proposed new features for the AWS Console - managing authorization policies for Cognito user groups using Amazon Verified Permissions.

## Overview

This application provides a UI for managing Cedar policies stored in Amazon Verified Permissions (AVP). It simulates how the AWS Cognito Console could allow users to:

- View authorization policies for a Cognito user group
- Add new authorization policies
- Edit existing policies
- Delete policies

The UI translates user-friendly form inputs into Cedar policy language and stores them in AVP.

## Configuration

### AWS Credentials

The backend server reads AWS credentials from `~/.aws/credentials`. Ensure your credentials file contains:

```
[default]
aws_access_key_id = YOUR_ACCESS_KEY
aws_secret_access_key = YOUR_SECRET_KEY
aws_session_token = YOUR_SESSION_TOKEN  # if using temporary credentials
```

The server also supports non-standard formats (uppercase keys, leading spaces) commonly generated by credential helper tools.

### Policy Store

To use a different policy store, edit `server.js`:

```javascript
const POLICY_STORE_ID = 'your-policy-store-id';
const REGION = 'us-east-1';  // change if needed
```

## Usage

### Group Detail Page

The main page displays:
- Group information (name, description, timestamps)
- Group members table (placeholder)
- Authorization policies table

From here you can:
- Click **Add authorization policy** to create a new policy
- Select a policy and click **Edit** to modify it
- Select a policy and click **Delete** to remove it

### Add/Edit Policy

The policy form includes:
- **Policy name**: Unique identifier (stored as `@id` annotation in Cedar)
- **Description**: Human-readable explanation (stored in AVP policy metadata)
- **Resource type**: Dropdown populated from the AVP schema
- **Actions**: Multi-select of actions applicable to the selected resource type
- **Attribute conditions**: Optional conditions comparing user and resource attributes

## API Endpoints

The backend server exposes these endpoints:

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/schema` | Fetch Cedar schema from AVP |
| GET | `/api/policies?groupName=X&namespace=Y` | List policies for a group |
| GET | `/api/policies/:policyId` | Get a single policy |
| POST | `/api/policies` | Create a new policy |
| DELETE | `/api/policies/:policyId` | Delete a policy |

## Policy Translation

The application translates form inputs to Cedar policies. For example:

**Form Input:**
- Policy name: `read-contracts`
- Description: `Allow reviewing contracts`
- Resource type: `Contract`
- Actions: `REVIEW`, `APPROVE`
- Conditions: `department = owner_department`

**Generated Cedar Policy:**
```cedar
@id("read-contracts")
permit (
    principal in NAMESPACE::CognitoGroup::"team-project-rosie",
    action in [NAMESPACE::Action::"REVIEW", NAMESPACE::Action::"APPROVE"],
    resource is NAMESPACE::Contract
)
when {
    principal.department == resource.owner_department
};
```

See `docs/policy-translation.md` for detailed translation rules.

## Troubleshooting

### "Could not load credentials from any providers"

1. Check that `~/.aws/credentials` exists and contains valid credentials
2. Ensure credentials are not expired (session tokens typically last 1-12 hours)
3. Restart the backend server after updating credentials

### "Error loading schema"

1. Verify the policy store ID in `server.js` is correct
2. Check that your AWS credentials have `verifiedpermissions:GetSchema` permission
3. Ensure the policy store has a schema defined

### Port already in use

If port 3002 or 5176 is in use:
```bash
# Find and kill the process
netstat -ano | grep 3002
taskkill /PID <pid> /F
```
